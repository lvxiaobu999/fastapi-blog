# 开发流程与模块说明

本文记录项目当前已经实现的结构、关键文件用途和功能开发步骤。每次新增或调整功能时都应同步更新本文，避免代码已经变化而文档仍描述旧行为。

## 当前请求分层

```text
HTTP 请求
  → Router：解析参数、注入依赖、声明状态码、转换 HTTP 异常
  → Schema：校验请求并限制响应字段
  → Service：执行业务规则、查询、密码处理和事务提交
  → Model：描述 ORM 字段与数据库约束
  → Session / Engine：管理事务和数据库连接
  → SQLite（开发）或 PostgreSQL（生产）
```

Router 不直接编写 SQL，Service 不处理 HTTP 响应，Model 不负责请求参数校验。保持边界清晰可以让业务逻辑独立测试，也方便未来加入认证和权限控制。

## JWT 认证与 AJAX 表单

登录调用链为 `layout.html/login.html → auth.js → api.js → POST /api/auth/token → api_auth.py → services/auth.py → users 表`。成功后前端把短期 Token 保存到 `localStorage`。

发帖调用链为 `post_form.html → posts.js → api.js（Bearer Header）→ api_posts.py → dependencies/auth.py → services/posts.py → posts 表`。新增、修改和删除帖子仅允许管理员；用户资料修改和删除仅允许本人或管理员；新增帖子的 `user_id` 来自 JWT 对应数据库用户，不接受客户端指定。右上角登录和注册按钮只打开 Bootstrap 模态框，独立 `/login`、`/register` 页面仍作为可访问的备用入口。

管理员通过交互式管理命令创建，密码输入不会回显：

```powershell
uv run python -m app.cli create-admin
```

命令使用当前 `ENV` 对应的数据库，运行前应确认环境和迁移已经就绪。公开 `/api/users` 注册接口始终创建普通用户，不能通过请求体传入 `is_admin`。

本次新增或修改的关键文件：

| 文件 | 用途 |
| --- | --- |
| `app/schemas/auth.py` | Bearer Token 响应契约 |
| `app/services/auth.py` | 显式封装密码哈希/校验与 JWT 创建/验证四个安全函数 |
| `app/dependencies/auth.py` | 当前用户和管理员依赖，统一 401/403 |
| `app/routers/api_auth.py` | OAuth2 表单登录接口 |
| `app/routers/api_posts.py` | 要求管理员 Token 才能新增、编辑或删除帖子 |
| `app/cli.py` | 交互式创建管理员，复用用户 Service 和 Argon2 哈希 |
| `app/static/js/api.js` | 公共 `jquery.ajax` 与 Bearer Header 封装 |
| `app/static/js/auth.js` | 登录、注册和退出交互 |
| `app/static/js/posts.js` | 发布、编辑帖子交互 |
| `app/static/js/forms.js` | 搜索和个人资料表单 AJAX 交互 |
| `app/templates/layout.html` | 导航认证模态框和 ES module 入口 |
| `tests/test_auth.py` | JWT 与管理员发帖授权回归测试 |

本功能不改变数据库结构，不需要新增 Alembic Migration。

## 帖子富文本编辑器

帖子编辑页使用 Toast UI Editor，正文以 Markdown 字符串保存到现有 `posts.content`，
因此没有数据库结构变化。编辑器支持所见即所得、Markdown、图片和 fenced code block；
JSON、JavaScript、Bash 示例分别使用 `json`、`javascript`、`bash` 语言标记。

图片调用 `POST /api/posts/images` 上传，仅管理员可用。后端按文件魔数允许 PNG、JPEG、
GIF、WebP，单个文件不超过 5 MB，保存到 `app/media/post_images/`，正文只保存随机文件名
对应的 `/media/post_images/...` URL。相关文件包括：

| 文件 | 用途 |
| --- | --- |
| `app/services/images.py` | 图片大小、文件签名和随机文件名校验 |
| `app/routers/api_posts.py` | 管理员图片上传接口 |
| `app/static/js/posts.js` | 初始化编辑器、上传图片和提交 Markdown |
| `app/templates/post_form.html` | 编辑器容器和 CDN 资源 |
| `app/templates/post.html` | 使用 Toast UI Viewer 渲染正文和代码块 |

VSCode 调试 JWT 管理员链路请阅读 `docs/VSCode调试JWT管理员链路.md`，该文按登录签发、
Bearer Header、Token 验证、数据库用户查询、`require_admin` 和发帖 Router 的顺序说明断点与变量观察方法。

## 应用与配置模块

| 文件 | 用途 |
| --- | --- |
| `app/main.py` | 创建 FastAPI 应用、挂载静态与媒体目录、注册 Router、处理全局 404 |
| `app/core/config.py` | 按 `ENV` 加载公共及环境专用配置，校验数据库类型和安全配置 |
| `app/templating.py` | 创建共享的 Jinja2 模板对象，并使用稳定的模板目录路径 |
| `.env` | 本机公共配置与 `SECRET_KEY`；已从 Git 跟踪中移除 |
| `.env.development` | 开发环境配置，使用 SQLite |
| `.env.production` | 生产环境非敏感配置；数据库凭据应由部署环境注入 |

配置文件自身已添加用途说明。`alembic.ini` 是例外：Alembic 在当前 Windows 环境按系统 GBK 编码读取 INI，直接写入 UTF-8 中文会导致迁移命令解析失败，因此该文件保留 ASCII 注释，对应中文解释如下：

- `script_location`：指定迁移脚本目录，`%(here)s` 表示以 `alembic.ini` 所在目录为基准。
- `prepend_sys_path`：把项目根目录加入模块搜索路径，使迁移环境能够导入 `app`。
- `sqlalchemy.url`：故意不写入 INI，由 `migrations/env.py` 从 Settings 读取，避免提交数据库凭据。
- `[loggers]` 及后续配置：只控制 Alembic CLI 日志，不影响 FastAPI 应用日志。

## 数据库模块

| 文件 | 用途 |
| --- | --- |
| `app/db/base.py` | 定义所有 ORM 模型共用的 `DeclarativeBase` 和 `MetaData` |
| `app/db/session.py` | 创建 AsyncEngine、`AsyncSessionLocal` 和 FastAPI 的异步 `get_db()` 依赖 |
| `app/models/user.py` | 定义 `users` 表字段、唯一约束、默认昵称、管理员标记和头像 URL |
| `alembic.ini` | Alembic 命令入口配置，不保存数据库密码 |
| `migrations/env.py` | 从 Settings 读取数据库地址，并把模型 MetaData 提供给 Alembic |
| `migrations/versions/20260722_01_create_users_table.py` | 首次创建 `users` 表的迁移，可通过 downgrade 删除该表 |

数据库结构变更不能只修改 Model。正常流程是修改 Model 后创建一个新的迁移文件，检查 `upgrade()` 和 `downgrade()`，再迁移目标环境。

```powershell
uv run alembic upgrade head
```

生产环境执行迁移前必须确认 `ENV=production` 和目标数据库，不能把开发 SQLite 当作生产迁移结果。

### 创建用户调用链

```text
POST /api/users
  → UserCreate 校验 username、email、password
  → api_users.create_user() 注入 Session
  → services.users.create_user()
      → 规范化邮箱
      → 检查用户名与邮箱
      → 使用 Argon2 哈希密码
      → INSERT + COMMIT + REFRESH
  → UserResponse 过滤 password 和 hashed_password
  → 201 Created
```

数据库唯一约束仍是并发安全的最终保证。Service 中的预查询用于提前给出友好错误，`IntegrityError` 捕获用于处理两个并发请求同时通过预查询的情况。

### 当前接口

```text
POST   /api/users             创建用户
GET    /api/users             分页列出用户
GET    /api/users/{user_id}   查询用户
PATCH  /api/users/{user_id}   部分更新用户
DELETE /api/users/{user_id}   删除用户
```

当前尚未实现 JWT 和认证依赖，因此用户接口暂时公开，只适合开发环境。文档不能把未来计划中的认证描述成已经具备的能力。

JWT 与认证的待开发技术方案见 `docs/JWT与用户认证开发流程.md`。该文件是实施计划，认证完成并通过测试前不能将其中的权限能力视为项目现状。

## 新功能开发步骤

1. 明确请求字段、响应字段、状态码和不在本次范围内的功能。
2. 阅读相关 Router、Schema、Service、Model、迁移和测试，确认完整调用链。
3. 先更新 Schema 与 Model 设计，确保字段长度、空值和唯一约束一致。
4. 数据库结构有变化时新增 Alembic Migration，不修改已应用的旧迁移。
5. 在 Service 实现业务规则和事务，在 Router 中只处理 HTTP 行为。
6. 为新模块添加中文模块 docstring，为公共类、函数、端点和复杂配置添加中文说明。
7. 使用隔离测试数据库覆盖成功路径及相关的冲突、404、422 等失败路径。
8. 更新接口文档、数据库文档和本文的文件用途及调用链。
9. 执行静态检查、目标测试、完整测试和迁移检查，再报告结果。

## 验证命令

```powershell
uv run ruff check app migrations tests
uv run pytest -q
uv run alembic current
uv run alembic check
```

测试必须通过 `app.dependency_overrides` 替换 `get_db`，避免操作开发或生产数据。`alembic check` 用于确认 ORM MetaData 是否还有未生成的迁移差异。

## 文档更新要求

每次交付至少检查以下内容：

- 新增、修改或删除了哪些关键文件。
- 每个文件为什么存在，归属哪一层。
- 请求从 Router 到数据库的调用链是否变化。
- 是否改变公开 API、环境配置或数据库结构。
- 新增迁移的升级和回退行为。
- 实际执行了哪些验证命令，哪些检查没有执行及原因。

代码中的注释用于解释局部设计和参数副作用；本文用于解释模块关系和完整开发过程，两者不能互相替代。
