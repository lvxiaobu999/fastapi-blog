# 数据库迁移与常用操作

本文说明项目如何使用 Alembic 管理 SQLite 和 PostgreSQL 表结构，以及开发过程中常用的数据库检查、备份和恢复操作。

## 一、本次帖子表使用了什么命令

生成 `posts` 表迁移文件时执行的是：

```powershell
uv run alembic revision --autogenerate -m "create posts table"
```

命令各部分含义：

```text
uv run
  使用当前项目由 uv 管理的虚拟环境执行命令

alembic revision
  创建一个新的迁移版本文件

--autogenerate
  比较 SQLAlchemy Model 与当前数据库结构，自动生成结构差异

-m "create posts table"
  为迁移添加说明，方便阅读迁移历史
```

该命令生成了：

```text
migrations/versions/f9892737bfc8_create_posts_table.py
```

迁移文件中的 `upgrade()` 包含：

```text
创建 posts 表
创建 posts.user_id 外键
创建 ix_posts_user_id 索引
```

`downgrade()` 包含相反操作：

```text
删除 ix_posts_user_id 索引
删除 posts 表
```

重要区别：`revision --autogenerate` 只是生成迁移文件，不会立即修改数据库。

## 二、`alembic upgrade head` 有什么用

执行迁移使用：

```powershell
uv run alembic upgrade head
```

它的含义是：

```text
读取当前数据库记录的迁移版本
        ↓
找到当前版本之后尚未执行的迁移
        ↓
按照迁移依赖顺序执行 upgrade()
        ↓
将数据库升级到最新版本 head
        ↓
更新 alembic_version 表中的版本号
```

在本项目中，这条命令把开发 SQLite 从：

```text
20260722_01   只有 users 表
```

升级到：

```text
f9892737bfc8  增加 posts 表和 user_id 索引
```

`head` 表示当前迁移链最前端的最新版本，不是数据库中的最新一条业务数据。

## 三、Alembic 有什么用

Alembic 是 SQLAlchemy 项目的数据库结构版本管理工具，可以理解为“数据库表结构的 Git 历史”。

Git 管理代码变化：

```text
旧代码 → commit → 新代码
```

Alembic 管理数据库结构变化：

```text
users 表 → migration → users + posts 表
```

它主要解决以下问题：

1. 记录每次表结构为什么变化。
2. 按顺序在开发、测试和生产数据库应用相同结构。
3. 知道某个数据库当前处于哪个结构版本。
4. 在经过确认后回退指定迁移。
5. 比较 ORM Model 和数据库，发现遗漏的迁移。
6. 部署新代码时可重复、可审查地升级数据库。

仅修改 SQLAlchemy Model 不会自动修改已有数据库。例如：

```python
class Post(Base):
    __tablename__ = "posts"
```

这只是在 Python 中描述表结构。要让已经存在的数据库真正拥有 `posts` 表，还需要迁移文件和 `alembic upgrade`。

## 四、项目中的 Alembic 文件

```text
alembic.ini
  Alembic CLI 基础配置

migrations/env.py
  加载 Settings、数据库地址和 Base.metadata

migrations/script.py.mako
  新迁移文件的生成模板

migrations/versions/
  按版本保存所有迁移文件

alembic_version
  Alembic 在目标数据库中创建的版本记录表
```

`alembic.ini` 没有保存数据库地址。`migrations/env.py` 会从当前环境的 Settings 读取 `DATABASE_URL`，避免把生产凭据提交到 Git。

## 五、标准迁移开发流程

### 第一步：确认当前环境

未设置 `ENV` 时默认开发环境：

```powershell
Remove-Item Env:ENV -ErrorAction SilentlyContinue
```

生产环境必须显式设置：

```powershell
$env:ENV = "production"
$env:DATABASE_URL = "postgresql+psycopg://<用户名>:<密码>@<主机>:5432/<数据库名>"
```

不要在终端日志、文档或提交记录中暴露真实生产连接地址。

### 第二步：修改 ORM Model

例如新增字段：

```python
summary: Mapped[str | None] = mapped_column(String(300), nullable=True)
```

Pydantic Schema、数据库约束和业务规则也要保持一致。

### 第三步：检查是否存在结构差异

```powershell
uv run alembic check
```

没有差异时输出：

```text
No new upgrade operations detected.
```

如果检测到新增表、字段或索引，说明 Model 和当前迁移历史不一致，需要创建迁移。

### 第四步：生成迁移

```powershell
uv run alembic revision --autogenerate -m "add post summary"
```

迁移说明应简短、明确，使用英文小写短语便于生成稳定文件名。

### 第五步：人工检查迁移文件

必须检查：

- `upgrade()` 是否只包含本次需要的变化。
- `downgrade()` 是否能合理回退。
- 是否意外删除表或字段。
- 字段类型、长度、空值和默认值是否正确。
- 外键、唯一约束和索引是否符合 Model。
- 新增非空字段时，历史数据如何获得有效值。

`--autogenerate` 是辅助工具，不保证生成结果一定符合业务意图。

### 第六步：备份并执行迁移

开发 SQLite 可以先停止应用并备份：

```powershell
Copy-Item fastapi_blog.db fastapi_blog.before_migration.db
```

然后执行：

```powershell
uv run alembic upgrade head
```

### 第七步：验证

```powershell
uv run alembic current
uv run alembic check
uv run pytest -q
```

## 六、常用 Alembic 命令

### 查看当前数据库版本

```powershell
uv run alembic current
```

当前项目正常结果应指向最新 head。

### 查看迁移历史

```powershell
uv run alembic history --verbose
```

简洁查看：

```powershell
uv run alembic history
```

### 查看最新迁移节点

```powershell
uv run alembic heads
```

正常的线性迁移通常只有一个 head。出现多个 head 说明产生了迁移分支，需要确认是否合并。

### 查看指定迁移内容

```powershell
uv run alembic show f9892737bfc8
```

### 升级到最新版本

```powershell
uv run alembic upgrade head
```

### 只向前执行一个版本

```powershell
uv run alembic upgrade +1
```

### 生成迁移文件

```powershell
uv run alembic revision --autogenerate -m "change description"
```

如果只是创建空迁移骨架：

```powershell
uv run alembic revision -m "data migration"
```

空迁移适合需要手写数据转换逻辑、而不是单纯表结构差异的情况。

### 生成 SQL 而不执行

```powershell
uv run alembic upgrade head --sql
```

这会把迁移 SQL 输出到终端，适合审查。输出可能包含表名和结构，但不应包含数据库密码。

### 回退一个版本

```powershell
uv run alembic downgrade -1
```

这是破坏性操作。当前帖子迁移的 downgrade 会直接删除 `posts` 表及其中全部帖子数据。执行前必须备份并确认目标数据库。

### 回退到指定版本

```powershell
uv run alembic downgrade 20260722_01
```

同样属于破坏性操作，不能仅为了测试命令而对需要保留的数据执行。

### 仅修改版本标记

```powershell
uv run alembic stamp head
```

`stamp` 只修改 `alembic_version`，不会执行建表或改字段 SQL。数据库实际结构已经由其他方式正确创建时才可使用；错误使用会造成“版本显示最新，但表结构缺失”。

## 七、SQLite 常用操作

### 确认数据库文件

开发配置：

```dotenv
DATABASE_URL=sqlite:///./fastapi_blog.db
```

对应项目根目录下的：

```text
fastapi_blog.db
```

该文件已被 `.gitignore` 忽略。

### 查看所有表

无需安装 sqlite3 CLI，可以使用 SQLAlchemy：

```powershell
uv run python -c "from sqlalchemy import inspect; from app.db.session import engine; print(inspect(engine).get_table_names())"
```

### 查看指定表字段

```powershell
uv run python -c "from sqlalchemy import inspect; from app.db.session import engine; print(inspect(engine).get_columns('posts'))"
```

字段输出可能较长，重点检查字段名、类型、空值和默认值。

### 查看索引

```powershell
uv run python -c "from sqlalchemy import inspect; from app.db.session import engine; print(inspect(engine).get_indexes('posts'))"
```

### 备份 SQLite

先停止正在写数据库的应用，再执行：

```powershell
Copy-Item fastapi_blog.db fastapi_blog.backup.db
```

直接复制一个正在写入的 SQLite 文件可能得到不一致备份。重要数据建议使用 SQLite 的备份 API 或专用备份工具。

### 恢复 SQLite 备份

恢复会覆盖当前数据库，属于破坏性操作。先确认路径并再次备份当前文件：

```powershell
Copy-Item fastapi_blog.db fastapi_blog.failed.db
Copy-Item fastapi_blog.backup.db fastapi_blog.db -Force
```

不要在应用仍然运行时恢复数据库。

## 八、PostgreSQL 常用操作

以下命令需要本机安装 PostgreSQL 客户端工具，并使用实际主机、用户和数据库名替换占位符。不要把密码直接写在命令历史中。

### 连接数据库

```powershell
psql -h <主机> -U <用户名> -d <数据库名>
```

### 在 psql 中查看表

```text
\dt
```

查看 `posts` 表结构：

```text
\d posts
```

查看当前连接：

```text
\conninfo
```

退出：

```text
\q
```

### 备份 PostgreSQL

推荐自定义格式：

```powershell
pg_dump -h <主机> -U <用户名> -d <数据库名> -F c -f fastapi_blog.backup
```

### 恢复 PostgreSQL

恢复会写入目标数据库，执行前必须确认环境：

```powershell
pg_restore -h <主机> -U <用户名> -d <目标数据库名> --clean fastapi_blog.backup
```

`--clean` 会先删除备份中已有对象，属于破坏性选项。生产恢复前应先在临时数据库演练并检查备份有效性。

## 九、SQLAlchemy Session 常用操作

### 查询

```python
from sqlalchemy import select

statement = select(Post).where(Post.id == post_id)
post = session.scalar(statement)
```

### 新增

```python
session.add(post)
session.commit()
session.refresh(post)
```

### 更新

```python
post.title = "新标题"
session.commit()
session.refresh(post)
```

### 删除

```python
session.delete(post)
session.commit()
```

### 处理可预期的写入异常

```python
try:
    session.commit()
except IntegrityError:
    session.rollback()
    raise
```

发生写入异常后，当前 Session 事务会处于失败状态；继续使用前必须 `rollback()`。

`commit()` 会持久化事务，`flush()` 只把当前变化同步到数据库连接但不提交事务，`refresh()` 会从数据库重新加载对象字段。

## 十、不要这样做

### 不要只修改 Model 而忽略迁移

```text
修改 Model ≠ 修改已有数据库
```

### 不要在生产中使用 create_all 代替迁移

```python
Base.metadata.create_all(engine)
```

它适合测试或一次性原型，但不会提供可靠的结构演进历史，也不会自动处理字段变更和数据迁移。

### 不要修改已经在其他环境执行过的旧迁移

应该新增迁移，而不是修改历史文件。否则不同环境会出现相同版本号对应不同结构的问题。

### 不要盲目信任 autogenerate

删除、重命名、非空字段和数据转换尤其需要人工检查。Alembic 可能把“字段重命名”识别成“删除旧字段并新增字段”，从而造成数据丢失。

### 不要在未确认环境时执行迁移

至少先运行：

```powershell
uv run alembic current
```

并确认 `ENV`、数据库类型和目标数据库符合预期。

```text

current
```
