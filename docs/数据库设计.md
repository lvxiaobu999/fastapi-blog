# 数据库设计

## 用户表 `users`

用户表负责保存登录凭据、公开资料和发帖权限。当前博客只有管理员能够发帖，因此暂时使用 `is_admin` 布尔字段即可，不需要提前引入角色表或完整的 RBAC 权限系统。

### 字段定义

| 字段 | SQLAlchemy 类型 | 是否为空 | 默认值 | 约束与用途 |
| --- | --- | --- | --- | --- |
| `id` | `Integer` | 否 | 数据库生成 | 主键，用户唯一标识 |
| `username` | `String(50)` | 否 | 无 | 唯一；用于登录和识别用户 |
| `email` | `String(254)` | 否 | 无 | 唯一；应用层保存前应去除首尾空格并转为小写 |
| `hashed_password` | `String(255)` | 否 | 无 | 只保存密码哈希，不保存明文，也不能出现在 API 响应中 |
| `nickname` | `String(50)` | 否 | 随机昵称 | 展示名称，不要求唯一，用户之后可以修改 |
| `image_file` | `String(200)` | 是 | `NULL` | 头像文件的存储名称；为空时使用默认头像 |
| `is_admin` | `Boolean` | 否 | `false` | 是否允许执行管理员操作，例如发布帖子 |

### 不保存 `image_path`

`image_path` 不应成为数据库字段，因为它可以由 `image_file` 和媒体访问前缀推导得到。重复保存会产生两个数据源：文件名或目录调整后，数据库中的旧路径可能失效。

模型通过只读属性生成公开 URL：

```python
@property
def image_path(self) -> str:
    if self.image_file:
        return f"/media/profile_pics/{self.image_file}"
    return "/static/images/default.jpg"
```

注意：这里返回的是浏览器访问 URL，不是服务器文件系统的真实路径。

### 头像文件命名

不建议使用 `username + 后缀名` 作为头像文件名，原因包括：

- 用户名修改后，文件名与用户不再一致。
- 覆盖同名文件时，浏览器或 CDN 可能继续使用旧缓存。
- 并发上传和文件后缀处理更容易出错。

上传服务应生成 UUID 等不可预测的存储名称，并在落盘前限制文件大小、允许的 MIME 类型和图片格式。防止磁盘耗尽依靠上传大小限制、配额和旧文件清理，而不是依靠覆盖文件名。

### 密码设计

注册请求中的 `password` 仅用于接收明文。Service 层必须使用项目已有的 `pwdlib[argon2]` 生成哈希，然后只把结果写入 `hashed_password`。

```text
请求 password → Service 生成 Argon2 哈希 → users.hashed_password
```

`hashed_password` 不属于 `UserResponse`。日志、异常信息和调试输出也不能记录明文密码或完整哈希。

### 用户名与邮箱唯一性

`username` 和 `email` 的唯一约束必须由数据库保证，不能只在接口中先查询。预查询可以提供友好的错误提示，但并发请求仍可能同时通过预检查，最终应捕获数据库唯一约束异常并转换成合适的业务错误。

邮箱应在 Service 层统一执行：

```python
normalized_email = email.strip().lower()
```

这可以避免同一邮箱因为大小写不同而重复注册。SQLite 与 PostgreSQL 的大小写比较行为存在差异，因此应用层规范化仍然必要。

### 管理员权限

字段使用 Python 风格的 `is_admin`，不使用 `isAdmin`。新用户必须默认是普通用户；注册接口不能允许客户端自行提交 `is_admin=true`。管理员身份应通过受控脚本、后台管理流程或数据库迁移设置。

当前只有“管理员”和“普通用户”两种权限状态，布尔字段足够。只有未来确实出现多角色、多权限组合时，再考虑拆分 `roles`、`permissions` 等表。

## 暂不增加的字段

当前需求没有明确要求账号禁用、邮箱验证、软删除或审计查询，因此暂不增加 `is_active`、`email_verified_at`、`deleted_at`、`created_at`、`updated_at` 等字段。需要相应业务功能时再引入，并通过 Alembic 迁移修改表结构。

## Schema 对应关系

- `UserCreate`：接收 `username`、`email`、`password`。
- `UserResponse`：返回 `id`、`username`、`email`、`nickname`、头像信息和 `is_admin`。
- `hashed_password`：仅存在于 ORM 模型内部，不能出现在响应 Schema。

Pydantic 和数据库的长度约束应保持一致：用户名最长 50，邮箱最长 254，昵称最长 50。密码的最大长度是请求层限制，不等同于哈希字段长度。

## 迁移说明

当前仓库尚未建立 Alembic 迁移目录，因此本次只更新设计、ORM 模型和 Schema，没有直接创建或修改数据库表。接入 Alembic 后，应通过新迁移创建 `users` 表；不要使用 `Base.metadata.create_all()` 代替生产迁移流程。
