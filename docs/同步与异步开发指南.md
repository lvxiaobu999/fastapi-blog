# 同步与异步开发指南

本文说明同步和异步代码的适用场景、项目完成异步化需要修改的配置、必须掌握的知识，以及当前 FastAPI + SQLAlchemy 异步调用链。

## 一、同步与异步解决什么问题

### 同步执行

同步代码执行一个操作时，当前线程会等待操作结束：

```text
请求 A 查询数据库
  → 线程等待数据库响应
  → 得到结果后继续
  → 返回响应
```

示例：

```python
def get_user(session: Session, user_id: int) -> User | None:
    return session.get(User, user_id)
```

FastAPI 中普通 `def` 路由会被放入线程池，因此数据库等待不会直接卡住主事件循环，但并发量会受到线程池大小和线程开销限制。

### 异步执行

异步代码等待数据库或网络时，会把事件循环让给其他任务：

```text
请求 A 等待数据库 ──┐
请求 B 处理参数     ├→ 同一个事件循环交替推进任务
请求 C 等待网络 ────┘
```

示例：

```python
async def get_user(session: AsyncSession, user_id: int) -> User | None:
    return await session.get(User, user_id)
```

异步不会让单条 SQL 本身执行得更快，它主要提高大量 I/O 等待场景下的并发利用率。

## 二、什么时候使用同步

适合同步代码的场景：

- 小型后台工具、一次性脚本和管理命令。
- 请求量较低，架构简单比高并发更重要。
- 使用的数据库或第三方库只有同步 API。
- 团队暂时不熟悉事件循环、协程和异步测试。
- 主要工作是 CPU 计算，而不是数据库和网络等待。

同步方案并不是落后方案。对于流量不高的博客，同步 SQLAlchemy 也完全可以工作，而且更容易调试。

## 三、什么时候使用异步

适合异步代码的场景：

- 同时存在大量数据库查询、HTTP 请求或文件网络 I/O。
- 接口需要调用多个远程服务。
- WebSocket、Server-Sent Events 或长连接。
- 高并发下希望减少等待线程数量。
- 使用的整个调用链都有可靠异步驱动。

本项目使用 FastAPI，并已经安装 `aiosqlite` 和支持异步的 `psycopg`，因此统一采用异步数据库调用可以作为学习和后续扩展基础。

## 四、异步不适合直接处理什么

异步不能自动解决 CPU 密集任务：

```text
密码哈希
图片压缩
视频转码
大型报表计算
机器学习推理
```

如果在 `async def` 中直接执行耗时 CPU 代码，事件循环仍然会被阻塞。

本项目的 Argon2 密码哈希使用工作线程：

```python
from app.services.auth import hash_password

hashed_password = await hash_password(password)
```

短时间 CPU 操作可以使用工作线程；长时间重计算应使用进程池、Celery、Arq 等后台任务系统。

## 五、本项目异步化修改了哪些配置

### 开发 SQLite URL

同步 URL：

```dotenv
DATABASE_URL=sqlite:///./fastapi_blog.db
```

异步 URL：

```dotenv
DATABASE_URL=sqlite+aiosqlite:///./fastapi_blog.db
```

区别在于 `+aiosqlite`，它告诉 SQLAlchemy 使用 SQLite 异步驱动。

### 生产 PostgreSQL URL

```dotenv
DATABASE_URL=postgresql+psycopg://<用户名>:<密码>@<主机>:5432/<数据库名>
```

SQLAlchemy 的 `psycopg` 方言可以配合 `create_async_engine()` 使用异步实现。真实生产连接地址只能由部署环境注入。

### Settings 校验

开发环境只接受：

```text
sqlite+aiosqlite://
```

生产环境只接受：

```text
postgresql+psycopg://
```

严格校验可以防止代码已经使用 AsyncEngine，却误传同步 SQLite URL。

## 六、异步 Engine 与 Session

当前数据库配置位于 `app/db/session.py`：

```python
engine = create_async_engine(
    settings.database_url,
    pool_pre_ping=True,
)

AsyncSessionLocal = async_sessionmaker(
    bind=engine,
    class_=AsyncSession,
    autoflush=False,
    expire_on_commit=False,
)
```

参数含义：

- `bind=engine`：Session 通过该异步 Engine 获取连接。
- `class_=AsyncSession`：工厂创建异步会话。
- `autoflush=False`：查询前不自动 flush，写入时机保持明确。
- `expire_on_commit=False`：提交后保留已加载属性，响应序列化不会隐式触发异步查询。
- `pool_pre_ping=True`：从连接池取连接前检测存活状态。

FastAPI 依赖：

```python
async def get_db() -> AsyncIterator[AsyncSession]:
    async with AsyncSessionLocal() as session:
        yield session
```

`async with` 在请求结束或异常时关闭 Session，并把连接归还连接池。

## 七、哪些操作需要 await

AsyncSession 中常见操作：

| 操作 | 是否 await | 示例 |
| --- | --- | --- |
| 添加对象 | 否 | `session.add(user)` |
| 按主键查询 | 是 | `await session.get(User, id)` |
| scalar 查询 | 是 | `await session.scalar(statement)` |
| scalars 查询 | 是 | `result = await session.scalars(statement)` |
| flush | 是 | `await session.flush()` |
| commit | 是 | `await session.commit()` |
| rollback | 是 | `await session.rollback()` |
| refresh | 是 | `await session.refresh(user)` |
| delete | 是 | `await session.delete(user)` |

`session.add()` 只修改 Session 的内存状态，不执行 I/O，所以不需要 `await`。其他可能访问数据库或触发级联加载的操作需要 `await`。

## 八、Service 如何改成异步

同步版本：

```python
def get_post(session: Session, post_id: int) -> Post | None:
    return session.scalar(select(Post).where(Post.id == post_id))
```

异步版本：

```python
async def get_post(session: AsyncSession, post_id: int) -> Post | None:
    statement = select(Post).where(Post.id == post_id)
    return await session.scalar(statement)
```

列表查询：

```python
async def list_posts(session: AsyncSession) -> list[Post]:
    result = await session.scalars(select(Post))
    return list(result)
```

写入事务：

```python
async def create_post(session: AsyncSession, post: Post) -> Post:
    session.add(post)
    await session.commit()
    await session.refresh(post)
    return post
```

可预期异常仍然需要回滚：

```python
try:
    await session.commit()
except IntegrityError:
    await session.rollback()
    raise
```

## 九、Router 如何改成异步

数据库依赖类型：

```python
DbSession = Annotated[AsyncSession, Depends(get_db)]
```

同步 Router：

```python
def get_post(post_id: int, session: DbSession):
    return post_service.get_post(session, post_id)
```

异步 Router：

```python
async def get_post(post_id: int, session: DbSession):
    return await post_service.get_post(session, post_id)
```

必须同时做到：

```text
Router 使用 async def
Service 使用 async def
调用 Service 时使用 await
Service 接收 AsyncSession
数据库 I/O 使用 await
```

只把 Router 改成 `async def`、内部仍调用同步数据库代码，反而会阻塞事件循环。

## 十、关联关系与异步懒加载

异步 ORM 应避免在响应序列化期间隐式懒加载：

```python
post.author.username
```

如果 `author` 没有预加载，属性访问可能尝试隐式执行 SQL，并产生 `MissingGreenlet`。

当前查询使用：

```python
statement = select(Post).options(selectinload(Post.author))
```

`selectinload` 会在 Service 的异步查询阶段加载作者，Router 和 Pydantic 序列化时只读取内存对象。

常用策略：

- `selectinload`：一对多和列表查询通常优先使用。
- `joinedload`：适合明确的一对一或多对一，但列表可能产生重复行。
- 不依赖响应阶段的默认 lazy load。

## 十一、Alembic 为什么也要调整

环境 URL 已改为异步驱动：

```text
sqlite+aiosqlite://
postgresql+psycopg://
```

原来的同步 `engine_from_config()` 不能直接使用异步驱动。当前 `migrations/env.py` 使用：

```python
async_engine_from_config(...)
```

并通过：

```python
await connection.run_sync(do_run_migrations)
```

让 Alembic 的同步迁移操作在异步连接上执行。CLI 命令保持不变：

```powershell
uv run alembic current
uv run alembic revision --autogenerate -m "change description"
uv run alembic upgrade head
uv run alembic check
```

## 十二、应用生命周期

应用退出时应释放连接池：

```python
@asynccontextmanager
async def lifespan(_app: FastAPI):
    yield
    await engine.dispose()
```

不要在应用启动时调用：

```python
Base.metadata.create_all()
```

正式表结构由 Alembic 管理。启动时自动建表会掩盖漏掉的迁移，使不同环境结构失去一致性。

## 十三、异步测试

测试数据库也使用异步驱动：

```python
engine = create_async_engine(
    "sqlite+aiosqlite://",
    poolclass=StaticPool,
)
```

通过 `run_sync` 创建隔离测试表：

```python
async with engine.begin() as connection:
    await connection.run_sync(Base.metadata.create_all)
```

异步 HTTP 测试使用：

```python
transport = ASGITransport(app=app)

async with AsyncClient(
    transport=transport,
    base_url="http://test",
) as client:
    response = await client.get("/api/posts")
```

测试函数标记：

```python
pytestmark = pytest.mark.anyio


async def test_list_posts(client: AsyncClient) -> None:
    response = await client.get("/api/posts")
    assert response.status_code == 200
```

测试通过 `dependency_overrides[get_db]` 注入 AsyncSession，不能把同步 Session 传给异步 Service。

## 十五、常见错误

### 忘记 await

错误：

```python
user = session.get(User, user_id)
```

此时得到的可能是协程对象，而不是 User。

正确：

```python
user = await session.get(User, user_id)
```

### Router 异步但 Service 同步

```python
async def route():
    return sync_database_query()
```

同步查询会阻塞事件循环。需要全链路异步，或明确放入线程池。

### 在异步响应阶段懒加载关系

未预加载 `post.author` 可能产生 `MissingGreenlet`。应在 Service 查询时使用 `selectinload`。

### 使用错误数据库 URL

```text
create_async_engine + sqlite:///...              错误
create_async_engine + sqlite+aiosqlite:///...   正确
```

### 把并发当成事务安全

异步只能提高 I/O 并发，不能替代数据库唯一约束、外键、事务隔离和锁。

## 十六、当前项目异步调用链


## 十七、选择建议

如果项目目标是尽快完成一个低流量个人博客，同步方案更简单，也足够可靠。

如果目标是学习 FastAPI 的异步生态、后续增加外部 API、WebSocket 或更高并发，当前统一异步方案更合适。

最重要的不是“异步一定更高级”，而是保持调用链一致：

```text
选择同步 → Engine、Session、Service、Router、测试都同步
选择异步 → AsyncEngine、AsyncSession、Service、Router、测试都异步
```

不要长期维护半同步半异步且边界不明确的数据库调用链。
